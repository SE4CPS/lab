Flower Class Structure (OOP View)

  • Flower class in OOP
  • Encapsulates (attribute) state: type, color, etc.
  • Flower class owns flower behavior: bloom(), grow()
  • Represents (real-world) object/thing

Flower Structure in Functional Programming

  • FP separates data and behavior
  • Flower is a plain object with no internal methods
  • Functions bloom(data), grow(data) defined externally
  • Data is immutable; each function returns a new version
  • "POJO": Plain Old JavaScript Object

What is OOP?

  • Encapsulates data and behavior inside objects
  • Objects have identity and manage their own state
  • Methods operate on internal properties via this

class Flower {
  constructor(type, color) {
    this.type = type;
    this.color = color;
    this.petals = 0;
  }
  bloom() {
    console.log(`${this.color} ${this.type} blooms!`);
  }
}

const flower1 = new Flower("rose", "red");
const flower1 = new Flower("tiger lily", "orange");

What is FP?

  • Separates data from behavior
  • Uses pure functions that take input and return output
  • Focuses on immutability and stateless logic
const flower = { type: "rose", color: "red", petals: 0 };
const bloom = flower => {
  console.log(`${flower.color} ${flower.type} blooms!`);
};

map / filter / reduce

  • map: transform each item
  • filter: select items based on condition
  • reduce: combine items into a single result
const flowers = [
  { type: "rose", petals: 30 },
  { type: "tulip", petals: 12 }
];

const totalPetals = flowers
  .filter(f => f.type === "rose")
  .map(f => f.petals * 2)
  .reduce((sum, p) => sum + p, 0);

Core Principles (OOP vs FP)

  • OOP: mutable state, object identity
  • FP: immutability, pure transformations
  • OOP uses this, FP returns new objects
// OOP
this.petals++;

// FP
{ ...flower, petals: flower.petals + 1 }

Behavior and State

  • OOP: methods modify internal object state
  • FP: functions accept state and return new state
  • State is explicit and passed in/out
// OOP
flower.grow();

// FP
grow(flower);

Thinking in Objects vs Functions

  • OOP: model real-world things (nouns)
  • FP: model transformations and logic (verbs)
  • OOP: behavior belongs to object
  • FP: behavior is external
// OOP
flower.bloom();

// FP
bloom(flower);

JS Support for Both

  • JavaScript is multi-paradigm
  • Supports class-based OOP and first-class functions
  • You can mix OOP and FP as needed
class Flower {}

const grow = flower => ({ ...flower, petals: flower.petals + 1 });

When to Use OOP or FP

  • OOP: identity, lifecycle, or encapsulation
  • FP: data processing, transformation, or stateless logic
  • Stakeholders, technology, and scope
// OOP
const rose = new Flower();
rose.bloom();

// FP
const rose = { type: "rose" };
bloom(rose);

Framework vs Library

JavaScript Framework
  • Full structure and control flow
  • Framework calls your code
  • Often handles routing, state, rendering
  • Examples: React, Angular, Vue, Express
  • Suited for building complete apps
JavaScript Library
  • Single-purpose utility collections
  • You call the library when needed
  • Focus on specific task: DOM, AJAX, etc.
  • Examples: jQuery, Lodash, Chart.js
  • Suited for enhancing parts of an app

Why Does Data Binding Matter?

Image source: HQ Software Lab, 2025

Why Does Data Binding Matter?

  • Keeps UI and data in sync (reactive updates)
  • Reduces manual updates to the DOM
  • Makes apps more responsive and interactive
  • Minimizes errors from inconsistent state
  • Saves development time and effort

Reactive: UI updates automatically whenever programâ€™s data changes.

Data Binding and State Management

  • One-way Binding: React
  • Two-way Binding: Angular, Vue

Data Binding Types

One-way Binding

Data flows in one direction: from component state to UI.

Example: React

Two-way Binding

Data flows both ways: UI updates state and state updates UI automatically.

Examples: Angular, Vue

Two-way Binding Without Framework


    document.addEventListener('DOMContentLoaded', function() {
        const input = document.getElementById('nameInput');
        const output = document.getElementById('output');
        function updateUI() {
            output.textContent = "Hello, " + input.value;
        }
        function programmaticallyUpdate() {
            input.value = "Mike";
            updateUI();
        }
        input.addEventListener('input', updateUI);  // Data flows from user input to program (then to UI)
        setTimeout(programmaticallyUpdate, 3000);    // Data flows from program to UI
    });

What is the Virtual DOM?

Image source: Medium / Code Wall

  • Virtual DOM is lightweight copy of the real DOM
  • When data changes, update Virtual DOM first
  • Compare the new and old Virtual DOM
  • Only changed parts are updated in the real DOM
  • This makes updates faster and more efficient

Why Not Just Use getElementById?

  • Manual DOM updates with getElementById do work for small apps
  • But in large apps, tracking what changed where becomes difficult
  • Frameworks with Virtual DOM and diffing handle this automatically
  • Less developer error, better performance for complex UIs
  • We write code once and framework handles the updates
