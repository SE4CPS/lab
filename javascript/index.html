Who Created JavaScript?

  • Invented in 1995
  • Created by Brendan Eich at Netscape
  • Originally called Mocha, then LiveScript
  • Renamed to JavaScript for marketing (Java was popular)

JavaScript Timeline

Year Event
1995 Brendan Eich creates JavaScript in 10 days at Netscape
1996 Renamed JavaScript, submitted to ECMA
1997 ECMAScript standard (ES1) is published
2009 ES5: JSON, strict mode, array extras
2015 ES6 (ES2015): classes, arrow functions, let/const, promises
2017–Now Annual releases: async/await (ES2017), modules, optional chaining, etc.

Regular Expressions: History

  • 1950s – Kleene defines regex in automata theory
  • 1960s – Used in Unix editors (ed, grep)
  • 1980s – Perl scripting popularizes regex
  • 2000s – Added to Java, JS, Python
  • Today – Standard across languages
// Example idea:
// (ab)* = "ab" repeated

What is a Regular Expression?

  • Pattern of characters
  • Defines search rules
  • Used for match, search, replace
/hello/   // matches "hello"

Literal Matching

  • Plain text = exact match
  • Case sensitive default
  • Use flags to modify
/cat/.test("cat");   // true
/cat/.test("Cat");   // false
/cat/i.test("Cat");  // true

Metacharacters

  • . – any character
  • \d – digit
  • \w – word char
  • \s – whitespace
/c.t/.test("cat");   // true
/\d/.test("abc1");   // true
/\s/.test("a b");    // true

Character Classes

  • [abc] – match a/b/c
  • [^abc] – not a/b/c
  • [0-9] – range of digits
/[abc]/.test("bat");    // true
/[^abc]/.test("dog");   // true
/[0-9]/.test("42");     // true

Quantifiers

  • * – 0 or more
  • + – 1 or more
  • ? – 0 or 1
  • {n,m} – range
/a*/.test("aaa");     // true
/a+/.test("b");       // false
/a{2,4}/.test("aaa"); // true

Anchors

  • ^ – start
  • $ – end
/^cat/.test("catnap");   // true
/cat$/.test("mycat");    // true

Groups

  • ( ) – capture
  • (?: ) – non-capture
  • \1 – backreference
/(ab)+/.test("abab");   // true
/(cat|dog)/.test("dog");// true

Alternation

  • | means OR
  • Combine with groups
/red|blue/.test("blue");// true
/(jpg|png)/.test("img.png");

Escaping

  • Use \ before special char
  • Else literal meaning
/\./.test("end.");   // true
/\$/.test("$100");   // true

Flags

  • i – ignore case
  • g – global
  • m – multiline
/cat/i.test("CAT");  // true
"cat cat".match(/cat/g); // ["cat","cat"]

Replace

  • Use .replace()
  • Supports regex
  • Groups supported
"2025-09-26".replace(/-/g, "/");
// "2025/09/26"

Regex test()

  • Returns true/false
  • Good for validation
/\d{5}/.test("95211");   // true
/\d{5}/.test("zip");     // false

Regex match()

  • Extracts matches
  • Returns array
"abc123".match(/\d+/);
// ["123"]

Regex split()

  • Split string by pattern
  • Useful for parsing
"a,b;c".split(/[,;]/);
// ["a","b","c"]

Lookahead

  • (?= ) positive
  • (?! ) negative
/\d(?=px)/.test("10px"); // true
/\d(?!px)/.test("10em"); // true

Email Validation

  • Check structure
  • Not foolproof
/^[\w.-]+@[\w.-]+\.\w+$/.test("me@pacific.edu");
// true

Phone Validation

  • Digits & dashes
  • Different formats
/^\d{3}-\d{3}-\d{4}$/.test("209-946-3072");
// true

Summary

  • Regex = pattern language
  • Use with match, test, replace
  • Great for validation & parsing
  • Powerful but tricky!
// Regex are everywhere!
// Use carefully & test often

map / filter / reduce

const flowers = [
  { type: 'rose', petals: 30 },
  { type: 'tulip', petals: 12 },
  { type: 'rose', petals: 20 }
];

const doubledRosePetals = flowers
  .filter(f => f.type === 'rose')
  .map(f => f.petals * 2);

const total = doubledRosePetals.reduce((sum, p) => sum + p, 0);
console.log(total);

Topic: map, filter, reduce (Agriculture)

Dataset: fields from a farm

const fields = [
  { crop: 'wheat', area: 12, yieldPerAcre: 3.1 },
  { crop: 'corn',  area: 20, yieldPerAcre: 4.2 },
  { crop: 'wheat', area: 8,  yieldPerAcre: 2.9 },
  { crop: 'rice',  area: 6,  yieldPerAcre: 5.0 },
];

// 1) filter: only wheat fields
const wheatFields = fields.filter(f => f.crop === 'wheat');

// 2) map: to total yield per field
const wheatYield = wheatFields.map(f => f.area * f.yieldPerAcre);

// 3) reduce: sum total wheat production
const totalWheat = wheatYield.reduce((sum, y) => sum + y, 0);
console.log(totalWheat, 'tons');

Think: data-in → transform → data-out. No manual index. No temp variables.

Agriculture: Pipeline vs Classic

Declarative (map/filter/reduce)
const totalWheat = fields
  .filter(f => f.crop === 'wheat')
  .map(f => f.area * f.yieldPerAcre)
  .reduce((s, v) => s + v, 0);
  • Reads top → down
  • Focus on what to do
  • Good for data dashboards, ETL
Imperative (if/else + loop)
let totalWheat2 = 0;
for (let i = 0; i < fields.length; i++) {
  const f = fields[i];
  if (f.crop === 'wheat') {
    const yieldTotal = f.area * f.yieldPerAcre;
    totalWheat2 += yieldTotal;
  }
}
  • Control how step-by-step
  • Easier to add debug logs
  • Good when flow is irregular

Topic Map: Data Pipelines vs Control Flow

Map / Filter / Reduce
  • Map → reshape field data
  • Filter → keep only crops we want
  • Reduce → compute total yield
  • Chainable → map().filter().reduce()

Used in: data lakes, ETL, analytics, reporting

If / Else / Loop
  • If → branch by crop or state
  • Else → fallback to default crop
  • Loop → walk every field
  • Nesting → allows irregular rules

Used in: UI flows, validation rules, error handling

Declarative vs Imperative

Declarative
  • Describe the result you want
  • Engine decides how
  • Shorter, composable
  • map/filter/reduce, SQL, React JSX
  • When: data transforms, predictable steps
Imperative
  • Describe every step
  • You control the loop, counters
  • Better for complex branching
  • for/while, if/else trees
  • When: orchestration, error paths, async flow

When to Use Which?

  • Use map/filter/reduce when you start from an array and want to transform/aggregate (agriculture totals, student grades, IoT sensor data).
  • Use if/else + loops when business logic is irregular (special cases, missing data, location-based rules, mixed data shapes).
  • Mix: filter with FP → then imperative for the “weird” 10%.

OOP vs FP (at a glance)

OOP FP
Data + behavior in objects Data is plain; behavior in functions
Mutable state, this Immutability, pure functions
Good for lifecycles, UI components Good for data pipelines

OOP Example

class Flower {
  constructor(type, color) {
    this.type = type;
    this.color = color;
    this.petals = 0;
  }
  bloom() {
    this.petals++;
    console.log(`${this.color} ${this.type} blooms! (${this.petals})`);
  }
}

const rose = new Flower('rose', 'red');
rose.bloom();

FP Example

const flowerData = { type: 'rose', color: 'red', petals: 0 };

const bloom = f => ({ ...f, petals: f.petals + 1 });
const show  = f => console.log(`${f.color} ${f.type} blooms! (${f.petals})`);

const after = bloom(flowerData);
show(after);

Backend JSON → Frontend: OOP or FP?

  • OOP: new Flower(json) for methods & lifecycles
  • FP: pass JSON through pure transformers
  • Pick by needs: identity/encapsulation vs. data processing

Framework vs Library

JavaScript Framework
  • Full structure and control flow
  • Framework calls your code
  • Often handles routing, state, rendering
  • Examples: React, Angular, Vue, Express
  • Suited for building complete apps
JavaScript Library
  • Single-purpose utility collections
  • You call the library when needed
  • Focus on specific task: DOM, AJAX, etc.
  • Examples: jQuery, Lodash, Chart.js
  • Suited for enhancing parts of an app

Why Does Data Binding Matter?

  • Keeps UI and data in sync (reactive updates)
  • Reduces manual updates to the DOM
  • Makes apps more responsive and interactive
  • Minimizes errors from inconsistent state

Reactive: UI updates automatically when data changes.

Data Binding Types

One-way Binding

Data flows in one direction: state → UI.

Example: React

Two-way Binding

Data flows both ways: UI ↔ state automatically.

Examples: Angular, Vue

One-way Binding Without Framework

document.addEventListener('DOMContentLoaded', () => {
  const input = document.getElementById('nameInput');
  const output = document.getElementById('output');
  input.addEventListener('input', () => {
    output.textContent = 'Hello, ' + input.value;
  });
});

<input id="nameInput" type="text" placeholder="Enter your name">
<h1 id="output">Hello, Mike</h1>

Practical Example: One-way Binding (React)

function App() {
  const [name, setName] = React.useState('Mike');
  return (
    <div>
      <input value={name} onChange={e => setName(e.target.value)} />
      <h1>Hello, {name}</h1>
    </div>
  );
}

Two-way Binding Without Framework

document.addEventListener('DOMContentLoaded', () => {
  const input = document.getElementById('nameInput');
  const output = document.getElementById('output');
  function updateUI() {
    output.textContent = 'Hello, ' + input.value;
  }
  function programmaticallyUpdate() {
    input.value = 'Mike';
    updateUI();
  }
  input.addEventListener('input', updateUI);   // User → program → UI
  setTimeout(programmaticallyUpdate, 3000);    // Program → UI
});

<input id="nameInput" type="text" placeholder="Enter your name">
<h1 id="output">Hello, Mike</h1>

Two-way Binding With Framework (Vue)

<template>
  <div>
    <input v-model="name" placeholder="Enter your name" />
    <h1>Hello, {{ name }}</h1>
  </div>
</template>
<script>
export default {
  data() { return { name: 'Mike' } },
  mounted() {
    setTimeout(() => { this.name = 'Programmatic Mike' }, 3000);
  }
}
</script>

Line-by-Line: Vue 3 Two-way

<script setup>
import { ref } from 'vue'
const msg = ref('Hello World!')
</script>
<template>
  <h1>{{ msg }}</h1>
  <input v-model="msg" />
</template>
  • Program → UI: Updating msg changes the heading.
  • User → Program: Typing updates msg.

What is the Virtual DOM?

Image source: Medium / Code Wall

  • Virtual DOM is a lightweight copy of the DOM
  • On changes, compare old vs new virtual tree
  • Patch only the changed nodes in the real DOM
  • Fewer layout thrashes → better performance

Why Not Just Use getElementById?

  • Manual DOM works for small apps
  • In larger apps, tracking what changed where is hard
  • Frameworks/VDOM handle diffs & updates reliably
  • Less error-prone; scales with complexity
