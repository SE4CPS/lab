Module 1
Introduction to Databases Management System

Before 1900: Manual Systems

  • Before 1440 - Manual Writing: Information was written by hand, making copying and dissemination extremely slow and labor-intensive.
  • 1440 - Printing Press: Invented by Johannes Gutenberg, it enabled faster copying and dissemination of written information.
  • 1800s - Shipping Letters: Physical mail was sent by ship. A letter from here to Hawaii could take up to two weeks to arrive.

20th Century: The Digital Revolution

  • 1970 - Relational Databases: Edgar F. Codd introduced the relational model, which became the foundation of modern database management systems (DBMS).
  • 1980s - SQL Standardization: Structured Query Language (SQL) became the standardized way to interact with relational databases.
  • 1990s - Client-Server Databases: Systems like Oracle and Microsoft SQL Server facilitated centralized databases accessible by multiple users over a network.

21st Century: Connectivity and Real-Time Revolution

  • 2000s - NoSQL Databases: Non-relational databases like MongoDB and Cassandra emerged to handle unstructured and semi-structured data.
  • 2010s - Cloud Databases: Platforms like AWS and Azure enabled scalable, distributed databases accessible over the internet.
  • 2010s - Big Data: Systems like Hadoop and Spark allowed storage and processing of massive datasets across distributed environments.
  • 2020s - AI Integration: Artificial intelligence began enhancing query optimization, data analysis, and automated insights in databases.
  • 2020s - Edge Databases: Localized databases for IoT and real-time applications reduced latency and enhanced data processing efficiency.

What is Databases Management System 🤔

Databases solve problems of managing, storing, and retrieving vast amounts of data efficiently.

  • Why centralize data? To avoid duplication and inconsistencies.
  • Why organize data? To make access faster and more reliable.
  • Why automate data management? To save time and reduce errors.

Why Databases Management System 🤔

More reasons

  • Explosion of Data: The amount of data generated globally is growing exponentially, requiring systems that can store, manage, and retrieve it efficiently.
  • Core Value Across Domains: Data is the foundation of industries like healthcare, finance, e-commerce, and social media, driving insights and innovation.
  • Reliability and Efficiency: DBMS ensure data integrity, availability, and security in critical applications.
  • Provides a foundation for understanding unstructured data and the evolution of NoSQL databases

Why Use Minimum Relationships?

  • Performance: Fewer relationships mean faster queries.
  • Simplicity: Easier to design, maintain, and understand.
  • Storage Efficiency: Reduces unnecessary data duplication.
  • Query Optimization: Minimizes complex joins and indexing overhead.
  • Consistency: Less chance of data inconsistencies.

Relationship Types (Cardinality)

1:1

Each record links to one record.

Entities: 1

1:N

One record links to many records.

Entities: 2

N:N

Many records link to many records.

Entities: 3

When to Use Relationship Type?

Cardinality When to Use Example
1:1 When each record in one table corresponds to exactly one record in another table. User and Profile
1:N When one record can be associated with multiple records in another table. Library and Books
N:N When multiple records in both tables can be related to each other. Students and Courses

Implement Relationships in SQL

libraryId INTEGER FOREIGN KEY REFERENCES Library(libraryId) -- establishes a one-to-many relationship:

  • One Library: A single library can have multiple books.
  • Many Books: Each book belongs to exactly one library.
  • Foreign Key: Ensures referential integrity by linking Book.libraryId to Library.libraryId.
  • Cardinality: 1:N (One library to many books).

Module 6
Relational Database Normalization

Table Normalization: 1NF to 3NF

  • First Normal Form (1NF): Eliminate duplicate columns and ensure all values are atomic (no repeating groups or arrays).
  • Second Normal Form (2NF): Ensure the table is in 1NF and all non-key attributes depend entirely on the primary key.
  • Third Normal Form (3NF): Ensure the table is in 2NF and remove transitive dependencies (non-key attributes should only depend on the primary key).

Relational Database Table Indexing

Indexing in Databases

  • Definition: A data structure that improves the speed of data retrieval operations.
  • Purpose: Allows quick lookups and reduces the need for full table scans.
  • Types of Indexes:
    • Primary Index: Automatically created on the primary key.
    • Unique Index: Ensures no duplicate values in a column.
    • Clustered Index: Sorts and stores rows in the table based on key values.
    • Non-Clustered Index: Points to the data rather than storing it in sorted order.
  • Trade-Off: Speeds up reads but can slow down writes (e.g., INSERT, UPDATE).

Relational Database Transactions

Transactions and ACID Properties

  • Transaction: Sequence of database operations that are executed as a single logical unit.
  • ACID Properties: Ensure reliability and consistency in transactions:
    • Atomicity: All operations in a transaction are completed, or none are applied.
    • Consistency: Transactions transform the database from one valid state to another.
    • Isolation: Concurrent transactions do not interfere with each other.
    • Durability: After transaction is committed, changes are permanent.

Access Control and Encryption

  • Access Control: Regulates who can access or modify data (e.g., roles like Admin, Viewer).
  • Encryption: Secures data by converting it into an unreadable format.

Relational Database and Data Distribution

Distributed SQL Databases and CAP Theorem

  • Distributed SQL Database: A database system spread across multiple nodes
  • CAP Theorem: In a distributed system, only two of the three can be guaranteed at a time:
    • Consistency: All nodes see the same data at the same time.
    • Availability: Every request receives a response, even during failures.
    • Partition Tolerance: The system continues to operate despite network partitions.

Data Aggregation in Data Warehouses

  • Definition: The process of combining and summarizing data from multiple sources.
  • Purpose: Provides high-level insights for analysis and decision-making.
  • Common Aggregations:
    • Sum, Average, Count
    • Maximum, Minimum
    • Group By dimensions (e.g., region, time, product)
  • Use Case: Monthly sales totals by region for strategic planning.

Course Description

A Database Management System (DMS) is designed for the efficient storage, access, and update of data. This course explores DMS from two main perspectives:

  • User-Centered Perspective: Data models, query languages, and design techniques.
  • System Implementation Perspective: Policies, algorithms, and data structures for DMS design and implementation.

Course Objectives

  • Understand DMS functionality to select and utilize a DMS effectively.
  • Analyze data requirements and develop database schemas.
  • Create, manipulate, and query data using SQL.
  • Understand transaction processing concepts (ACID).
  • Develop and implement access control policies.
  • Evaluate and utilize indexing structures.
  • Understand the theoretical foundations of query languages.
  • Integrate a DMS with a programming environment.
