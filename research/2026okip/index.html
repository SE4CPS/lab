Problem: Understanding Software Updates and Security Risks

  • Release notes are large and difficult to analyze.
  • Security vulnerabilities are spread across CVE and advisory feeds.
  • User-reported issues appear in Reddit and GitHub discussions.
  • Manual review is slow and often inconsistent.
Because of this fragmentation, it becomes difficult to quickly determine whether an update is safe to install, whether one or more versions have known vulnerabilities, and what issues users are already reporting.

Technical Contribution 1: Vendor Extraction

Why it helped

  • Maps user wording to canonical vendor names.
  • Handles aliases such as k8s → kubernetes and nodejs → node.
  • Improves retrieval precision before any API call is made.
  • Supports strict routing instead of loose keyword matching.
app.py · canonical vendor mapping

def extract_vendors(q: str):
    alias_map = {
        "k8s": "kubernetes",
        "docker": "docker",
        "postgresql": "postgresql",
        "linux": "linux",
        "windows": "windows",
        "nodejs": "node",
        "microsoft": "windows",
    }
    found = []
    for alias, canonical in alias_map.items():
        if re.search(rf"\b{re.escape(alias)}\b", q.lower()):
            found.append(canonical)
    return list(dict.fromkeys(found))

Technical Contribution 2: Time-Aware Query Parsing

Why it helped

  • Transforms vague phrases such as last week or this month into concrete date windows.
  • Reduces irrelevant historical results.
  • Supports contextual search instead of plain keyword matching.
app.py · natural-language time windows

def parse_time_window(q: str, now=None):
    if "last week" in q.lower():
        monday = (now - timedelta(days=now.weekday() + 7)).date()
        sunday = monday + timedelta(days=6)
        return monday, sunday

    if "this month" in q.lower():
        first = now.replace(day=1).date()
        last_day = calendar.monthrange(now.year, now.month)[1]
        return first, last_day

def determine_allowed_sources(query: str, vendors: list[str]):
    wants_security = any(term in query.lower() for term in SECURITY_HINTS)
    wants_release = is_release_query(query)

    return {
        "use_os": wants_release,
        "use_reddit": any(term in query.lower() for term in DISCUSSION_HINTS),
        "use_cisa": wants_security,
        "gh_release_repos": sorted(set(repos_for(vendors))),
    }

Technical Contribution 4: Realtime GitHub Release Fetching

Why it helped

  • Added live release evidence rather than relying only on imported static documents.
  • Helps answer recent vendor questions correctly.
  • Improves freshness of the final response.
app.py · GitHub release API

@st.cache_data(ttl=600)
def fetch_github_releases(repo: str, limit: int = 30):
    url = f"https://api.github.com/repos/{repo}/releases"
    r = requests.get(url, headers={"Accept": "application/vnd.github+json"}, timeout=10)
    if r.status_code == 200:
        return [
            {
                "repo": repo,
                "title": rel.get("name") or rel.get("tag_name"),
                "summary": (rel.get("body") or "")[:500],
                "published": rel.get("published_at"),
            }
            for rel in r.json()[:limit]
        ]

Lessons Learned

  • Combining RAG with realtime APIs improves situational awareness.
  • Semantic retrieval is stronger than plain keyword lookup.
  • Routing is essential to avoid unnecessary noise and cost.
  • Caching and query filtering improve system speed.
  • Fresh evidence matters when answering update and security questions.
  • Static document stores remain useful for background context.
  • The strongest answers come from disciplined fusion, not from one source alone.

Limitations and Future Work

Limitations

  • API latency can still affect responsiveness.
  • External sources may disagree or change format.
  • LLM interpretation still requires grounding safeguards.
  • Rate limits remain a practical constraint.

Future Work

  • More robust API routing and caching strategies
  • Additional security and package feeds
  • Larger-scale evaluation
  • Multi-agent orchestration and parallel fetching

Conclusion

  • Starting from a static RAG baseline, we improved the system with realtime evidence.
  • Intelligent routing and contextual query handling improved answer precision.
  • The result was 30% less manual inspection and 36% better correct-answer detection.
  • This prototype shows that software update intelligence benefits from a careful combination of established retrieval methods and realtime AI-assisted synthesis.
